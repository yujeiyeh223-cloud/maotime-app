# 卯時設計 套量服外借系統(maotime-app)

## 這是什麼

給同事在**手機現場**用的樣衣(套量服)外借管理 PWA:掃 QR 或手動選件 → 借出 / 歸還 / 預約,另有庫存總覽與儀表板。客戶借樣衣要當場簽名,所以流程設計成手機單手操作。

- 線上位址:`https://yujeiyeh223-cloud.github.io/maotime-app/`
- 資料主體:**Google Sheets**,透過 Google Apps Script(GAS)Web App 存取(`index.html` 裡的 `GAS_URL`)

## 與主系統(maoshi-system)的關係

**維持獨立,不合併、不改寫。** 這個 App 比主系統早存在、資料在自己的 Sheets 裡。

主系統(案件管理系統)對它是**唯讀整合**:主系統讀這裡的 GAS `getInventory` 呈現電腦版總覽與儀表板卡片,但**借還預約仍然只在這支手機 App 現場操作**(要簽名)。主系統那邊不寫入、也沒有「新增衣服」——加樣衣仍在 Google Sheets 手動維護。

所以:**不要**把借還功能搬進主系統,也不要在這裡加主系統已經有的東西。

## 架構

刻意極簡,整個 repo 只有三個檔案:

| 檔案 | 內容 |
|------|------|
| `index.html` | **整個 App**(HTML + CSS + JS 全部內嵌,約 1500 行)。畫面用 `goPage()` 切換,沒有前端框架 |
| `sw.js` | Service Worker,**網路優先**(每次先抓最新,失敗才用快取),讓 App 可離線開啟 |
| `apple-touch-icon.png` | 加到主畫面的圖示(180×180) |

外部相依只有 CDN 的 `html5-qrcode`(掃 QR 用)。

## 部署與改動規矩(重要)

- **沒有 PR / CI 流程。推上 `main` 就等於立刻部署到同事正在使用的正式 App。** 跟主系統(功能分支 → CI → 合併)完全不同 —— 改動前先跟業主確認,不要自行推上去。
- 改 `index.html` 時要升版本號,而且**三個地方必須同時改成同一個值**(v36 → v37…):

  | 位置 | 內容 |
  |------|------|
  | `sw.js` | `const CACHE_VERSION = 'v36'` |
  | `index.html` 約 1440 行 | `const CURRENT_VERSION = 'v36'` |
  | `index.html` 約 252 行 | 畫面上顯示的 `套量服外借管理系統 v36` |

  **只改其中一邊會出事**:`checkVersion()` 是抓 `sw.js` 跟 `index.html` 的 `CURRENT_VERSION` 比對,不一致就跳「發現新版本」橫幅——而且**永遠關不掉**(按「立即更新」重載後兩邊仍然不一致,又立刻跳出來)。2026-07-23 就是只升了 `sw.js` 而踩到這個坑。
- **不要**把新檔案加進 SW 的 `cache.addAll` 陣列:那個陣列只要有任一檔案 404,整個 Service Worker 安裝就會失敗、連離線功能一起壞掉。反正它是網路優先 + 執行時快取,新檔案第一次被抓到就會進快取。
- 沒有測試框架,改完靠**手機實機操作驗證**(掃碼、借出、歸還、預約、庫存各走一遍)。
- iOS 已經「加到主畫面」的捷徑,**改了圖示或名稱要刪掉重加才會更新**(加入當下就把圖示複製走了)。

## 本機開發

沒有建置流程,`index.html` 直接用瀏覽器開就能跑(GAS 是線上的,本機也連得到)。

```
git pull            # 動工前先拉,避免蓋掉別人的改動
# 改 index.html → 記得升 sw.js 的 CACHE_VERSION
git add -A && git commit && git push   # push = 上線,先確認過再推
```

**注意**:桌面上的 `套量服外借系統` 資料夾是 2026-07-15 的舊複本、不是 git repo,**不要拿它當來源**。正本一律以這個 repo 為準。
